Blockchain in This Project

Block Structure

Each block stores story text, author, timestamp, previous hash, and its own cryptographic hash. A block represents one accepted continuation.

Chain Formation

Blocks are linked by storing the previous block’s hash. Any modification breaks the chain, making history tamper-evident.

Immutability

Once a block is finalized, it is never modified or deleted. The story grows append-only for eternity.

Backend Architecture

The backend is implemented using Flask and Socket.IO.

Real-time communication is handled via WebSockets for instant updates.

CHAIN — The finalized immutable ledger of the story.

CANDIDATES — Temporary proposed continuations awaiting consensus.

STORIES — Archived historical chains of completed works.

Data Flow

User Input

Scribes submit story text from the frontend via Socket.IO events to the backend oracle.

Candidate Creation

Backend generates a candidate block using the last block’s hash and stores it in the pool.

Broadcast

All connected clients receive live cryptographic updates when candidates or votes change.

Voting & Consensus

Each candidate belongs to a specific previous block in the sequence.

Users vote in real time; duplicate voting is prevented via session IDs.

The candidate with the highest votes is selected for finalization.

If a tie occurs, votes are reset, encouraging further collaboration and discussion.

Block Finalization

Winner Selection

The most voted candidate is chosen. Ties are explicitly detected and handled by the protocol.

Hash Generation

A unique cryptographic hash is generated using current data and the ancestor's seal.

Chain Update

The block is appended to the chain, and candidates are purged from the temporal pool.

Persistence

New stories archive the current chain into the STORIES structure.

A new Genesis block initializes the next era of collaboration.

All state is saved to state.json after every update for durability.

This guarantees the manuscript survives even through server restarts.
